JOIN Essenziali

Durata: 25 minuti teoria + 25 minuti esercizi

4.1 Perché esistono i JOIN

Il problema dei database relazionali

Nei database relazionali, i dati sono distribuiti su più tabelle per evitare ridondanza e garantire l'integrità. Ma questo significa che per rispondere a domande di business, spesso devi combinare informazioni da tabelle diverse.

Immagina di avere queste due tabelle:

Tabella clienti

id_clientenomecitta
1Mario RossiMilano
2Anna BianchiRoma
3Luca VerdiNapoli

Tabella ordini

id_ordineid_clienteprodottototale
1011Laptop899
1022Mouse25
1031Monitor350

La domanda: "Qual è il nome del cliente che ha fatto l'ordine 101?"

Le informazioni sono in DUE tabelle diverse! Il nome e in clienti, gli ordini in ordini. Devi combinarle.

Analogia pratica

Immagina due fogli Excel: uno con l'elenco degli studenti (nome, matricola), uno con i voti degli esami (matricola, voto). Per sapere "che voto ha preso Mario?", devi guardare entrambi i fogli e collegare le righe tramite la matricola. Questo e esattamente cio che fa un JOIN.

La soluzione: JOIN

I JOIN combinano righe di tabelle diverse basandosi su una colonna in comune (solitamente una chiave).

SQL - Il tuo primo JOIN
SELECT clienti.nome, ordini.prodotto, ordini.totale
FROM clienti
JOIN ordini ON clienti.id_cliente = ordini.id_cliente;

Risultato:

nomeprodottototale
Mario RossiLaptop899
Anna BianchiMouse25
Mario RossiMonitor350

Il JOIN ha combinato le informazioni: ora vediamo il nome del cliente insieme ai dettagli dell'ordine!

4.2 Primary key e foreign key

Per capire come funzionano i JOIN, devi comprendere due concetti fondamentali:

Primary key (chiave primaria)

Una colonna (o combinazione di colonne) che identifica univocamente ogni riga di una tabella. Caratteristiche:

  • Valori unici - non si ripetono mai
  • Mai NULL - sempre presente
  • Solitamente chiamata id o id_tabella

Foreign key (chiave esterna)

Una colonna che fa riferimento alla primary key di un'altra tabella. Crea la "relazione" tra le tabelle. Caratteristiche:

  • Puo contenere valori duplicati
  • Puo essere NULL (se la relazione e opzionale)
  • Il valore deve esistere nella tabella referenziata
Tabella CLIENTI                    Tabella ORDINI
+-------------+                    +-------------+
| id_cliente  | <-- PRIMARY KEY    | id_ordine   | <-- PRIMARY KEY
| nome        |                    | id_cliente  | <-- FOREIGN KEY (riferisce clienti)
| citta       |                    | prodotto    |
+-------------+                    | totale      |
       ^                           +-------------+
       |                                  |
       +----------------------------------+
                    RELAZIONE
         Un cliente puo avere MOLTI ordini
                    (1 : N)

Tipi di relazioni

Tipo Descrizione Esempio
1:N (uno a molti) Un record nella tabella A puo avere molti record correlati in B Un cliente puo avere molti ordini
1:1 (uno a uno) Un record in A corrisponde a esattamente un record in B Un dipendente ha un solo badge
N:N (molti a molti) Molti record in A collegati a molti record in B Studenti e corsi (richiede tabella di collegamento)

4.3 INNER JOIN

INNER JOIN

Restituisce solo le righe che hanno corrispondenza in ENTRAMBE le tabelle. Se una riga non ha match, viene esclusa dal risultato.

    Tabella A          Tabella B
    +-------+          +-------+
    |   A   |          |   B   |
    |   +---+---+      |       |
    |   |INNER |      |       |
    |   | JOIN |------+       |
    |   +---+---+      |       |
    |       |          |       |
    +-------+          +-------+

    Il risultato contiene SOLO l'intersezione:
    righe che esistono in entrambe le tabelle

Sintassi

SQL - INNER JOIN
SELECT colonne
FROM tabella1
INNER JOIN tabella2 ON tabella1.chiave = tabella2.chiave;
Nota

INNER JOIN e JOIN sono equivalenti. La parola INNER è opzionale ma rende il codice più esplicito.

Esempio dettagliato

clienti

idnome
1Mario
2Anna
3Luca

ordini

id_ordineid_clientetotale
1011899
102225
1031350
SQL
SELECT clienti.nome, ordini.id_ordine, ordini.totale
FROM clienti
INNER JOIN ordini ON clienti.id = ordini.id_cliente;

Risultato:

nomeid_ordinetotale
Mario101899
Anna10225
Mario103350
Nota importante

Luca (id=3) NON appare nel risultato! Perché? Perché non ha ordini nella tabella ordini, quindi non c'è corrispondenza. INNER JOIN esclude le righe senza match.

4.4 LEFT JOIN e RIGHT JOIN

LEFT JOIN (o LEFT OUTER JOIN)

Restituisce TUTTE le righe della tabella sinistra (quella nel FROM), più le corrispondenze dalla tabella destra. Se non c'è match, le colonne della tabella destra saranno NULL.

    Tabella A          Tabella B
    +-------+          +-------+
    | XXXXX |          |       |
    | XXXXX +---+      |       |
    | XXXXX |   |------+       |
    | XXXXX +---+      |       |
    | XXXXX |          |       |
    +-------+          +-------+

    LEFT JOIN: tutta la tabella A + match da B
    Le X indicano le righe incluse (tutta la tabella A)

Sintassi LEFT JOIN

SQL
SELECT colonne
FROM tabella_sinistra
LEFT JOIN tabella_destra ON condizione;

Esempio LEFT JOIN

SQL - LEFT JOIN
SELECT clienti.nome, ordini.id_ordine, ordini.totale
FROM clienti
LEFT JOIN ordini ON clienti.id = ordini.id_cliente;

Risultato:

nomeid_ordinetotale
Mario101899
Mario103350
Anna10225
LucaNULLNULL

Luca appare! Anche se non ha ordini. Le colonne degli ordini sono NULL perché non ci sono corrispondenze.

RIGHT JOIN

RIGHT JOIN e l'opposto di LEFT JOIN: restituisce tutte le righe della tabella destra.

SQL - RIGHT JOIN
SELECT clienti.nome, ordini.id_ordine
FROM clienti
RIGHT JOIN ordini ON clienti.id = ordini.id_cliente;
Consiglio pratico

RIGHT JOIN e raramente usato nella pratica. Puoi sempre riscrivere un RIGHT JOIN come LEFT JOIN invertendo l'ordine delle tabelle. La maggior parte dei programmatori preferisce LEFT JOIN per consistenza.

Quando usare quale JOIN?

Tipo JOIN Quando usarlo Esempio d'uso
INNER JOIN Voglio solo dati completi, dove esiste corrispondenza Report ordini con dettagli cliente (solo ordini validi)
LEFT JOIN Voglio tutti i record della tabella principale + eventuali correlazioni Tutti i clienti e i loro ordini (anche clienti senza ordini)
RIGHT JOIN Raramente usato - inverti le tabelle e usa LEFT -

Caso d'uso importante: trovare mancanze

LEFT JOIN + WHERE IS NULL e un pattern comune per trovare "chi NON ha...":

SQL - Trovare clienti senza ordini
-- Clienti che NON hanno mai ordinato
SELECT clienti.nome, clienti.citta
FROM clienti
LEFT JOIN ordini ON clienti.id = ordini.id_cliente
WHERE ordini.id_ordine IS NULL;

Risultato: Solo "Luca" - l'unico cliente senza ordini.

4.5 La condizione ON

Cos'e ON

La clausola ON specifica come le due tabelle sono correlate, cioè quali colonne confrontare per trovare le corrispondenze.

Sintassi

SQL
FROM tabella1
JOIN tabella2 ON tabella1.colonna = tabella2.colonna

Condizioni ON comuni

SQL - Vari tipi di condizioni ON
-- Uguaglianza semplice (più comune)
ON clienti.id = ordini.id_cliente

-- Condizioni multiple (AND)
ON tabella1.col1 = tabella2.col1 
   AND tabella1.col2 = tabella2.col2

-- Esempio pratico con più condizioni
ON vendite.anno = budget.anno 
   AND vendite.regione = budget.regione

JOIN con più di due tabelle

Puoi concatenare più JOIN per combinare tre, quattro o più tabelle:

SQL - JOIN multipli
-- Combinare clienti, ordini e prodotti
SELECT 
    clienti.nome,
    ordini.id_ordine,
    prodotti.nome_prodotto,
    prodotti.categoria
FROM clienti
JOIN ordini ON clienti.id = ordini.id_cliente
JOIN prodotti ON ordini.id_prodotto = prodotti.id;

4.6 Alias di tabella

Perché usare alias per le tabelle?

Quando lavori con JOIN, devi spesso specificare tabella.colonna per evitare ambiguità. Con nomi di tabelle lunghi, il codice diventa verboso. Gli alias rendono il codice più pulito e leggibile.

Sintassi

SQL
-- Con AS (esplicito)
FROM clienti AS c
JOIN ordini AS o ON c.id = o.id_cliente

-- Senza AS (implicito - funziona lo stesso)
FROM clienti c
JOIN ordini o ON c.id = o.id_cliente

Confronto: con e senza alias

Senza alias (verboso)

SQL
SELECT 
    clienti.nome, 
    clienti.citta, 
    ordini.prodotto, 
    ordini.totale
FROM clienti
INNER JOIN ordini 
    ON clienti.id_cliente 
     = ordini.id_cliente
WHERE clienti.citta = 'Milano';

Con alias (pulito)

SQL
SELECT 
    c.nome, 
    c.citta, 
    o.prodotto, 
    o.totale
FROM clienti c
JOIN ordini o 
    ON c.id_cliente 
     = o.id_cliente
WHERE c.citta = 'Milano';

Alias obbligatori: self-join

Quando fai un JOIN di una tabella con se stessa, gli alias sono obbligatori per distinguere le due "copie" della tabella.

SQL - Self-Join
-- Trovare dipendenti e i loro manager (stessa tabella!)
SELECT 
    d.nome AS dipendente, 
    m.nome AS manager
FROM dipendenti d
LEFT JOIN dipendenti m ON d.id_manager = m.id;

In questo esempio, d rappresenta i dipendenti e m rappresenta i manager (che sono anche dipendenti). Senza alias, sarebbe impossibile distinguerli.

Soluzione

Esercizio 4.3

Trova i clienti che NON hanno mai fatto ordini.

Mostra Soluzione
Soluzione
SELECT c.nome, c.citta
FROM clienti c
LEFT JOIN ordini o ON c.id = o.id_cliente
WHERE o.id_ordine IS NULL;

Pattern importante: LEFT JOIN + WHERE IS NULL per trovare "mancanze".

Esercizio 4.4

Per ogni cliente di Milano, mostra il totale speso in tutti gli ordini.

Mostra Soluzione
Soluzione
SELECT c.nome, SUM(o.totale) AS totale_speso
FROM clienti c
JOIN ordini o ON c.id = o.id_cliente
WHERE c.citta = 'Milano'
GROUP BY c.nome;

Combiniamo JOIN con GROUP BY e SUM per calcolare il totale per cliente.

Esercizio 4.5

Mostra nome cliente, numero di ordini e totale speso per ogni cliente che ha fatto almeno 2 ordini.

Mostra Soluzione
Soluzione
SELECT 
    c.nome, 
    COUNT(o.id_ordine) AS num_ordini,
    SUM(o.totale) AS totale_speso
FROM clienti c
JOIN ordini o ON c.id = o.id_cliente
GROUP BY c.nome
HAVING COUNT(o.id_ordine) >= 2;

JOIN + GROUP BY + HAVING: la combinazione potente per analisi avanzate.

Esercizio 4.6 (Avanzato)

Mostra tutti gli ordini con nome cliente e citta, ordinati per data decrescente. Includi gli ordini anche se il cliente non esiste nel database (usa RIGHT JOIN).

Mostra Soluzione
Soluzione
SELECT c.nome, c.citta, o.prodotto, o.totale, o.data
FROM clienti c
RIGHT JOIN ordini o ON c.id = o.id_cliente
ORDER BY o.data DESC;

L'ordine 105 appare con NULL per nome e città perché il cliente 5 non esiste.

Cheat sheet finale

SQL - Template Query Completa
-- Query SQL completa con tutti gli elementi
SELECT 
    t1.colonna1,
    t2.colonna2,
    COUNT(*) AS conteggio,
    SUM(t1.valore) AS totale,
    AVG(t1.importo) AS media
FROM tabella1 t1
LEFT JOIN tabella2 t2 ON t1.id = t2.id_riferimento
WHERE t1.condizione = 'valore'
    AND t2.altra_condizione > 100
GROUP BY t1.colonna1, t2.colonna2
HAVING COUNT(*) > 1
ORDER BY totale DESC
LIMIT 10;
